# Layer Architecture

## Направление зависимостей

Строгое правило: каждый слой может импортировать только из слоёв правее.

```
app  →  features  →  support  →  kernel  →  lib
```

Импорт в обратном направлении запрещён. Например, `kernel` не может импортировать из `support`, а `support` — из `features`.

---

## Слои

### lib — библиотеки

Универсальные утилиты, не знающие о контексте проекта. Код в `lib` не имеет представления о том, что это мобильное приложение, что есть авторизация или какой-то конкретный API.

**Что здесь живёт:**
- Фабрики контекстов (`createApiContext`, `createReducerContext`) — generic-обёртки без привязки к конкретным типам или схемам
- Утилиты для React (`useStaticInitialize`) и RxJS (`useSubscribeSubject`)
- Базовые типы ошибок (`ApiError`, type guards)
- Константы по умолчанию (настройки query client)

**Правила:**
- Не импортирует ничего из проекта — только из `node_modules`
- Не знает о бизнес-домене, роутинге, схемах API
- Код переносим в другой проект без изменений

---

### kernel — ядро проекта

Базовые связующие модули, которые привязывают библиотеки из `lib` к контексту конкретного проекта. Kernel знает о проектных схемах, переменных окружения, конкретных сервисах, но не знает о бизнес-сценариях.

**Что здесь живёт:**
- **api/** — привязка `createApiContext` к конкретной OpenAPI-схеме (`paths`), создание fetch-клиентов с `baseUrl` из env, auth middleware с JWT
- **session/** — управление токенами через `expo-secure-store`, refresh-логика, хуки login/logout
- **ui/** — дизайн-система проекта: `Button`, `Text`, `Avatar`, `Input`, `ScreenLayout`, `Spinner` и т.д.
- **theme/** — привязка NativeWind к проекту
- **env.ts** — валидация переменных окружения через Zod

**Правила:**
- Импортирует только из `lib` и `node_modules`
- Не знает о конкретных бизнес-фичах (не импортирует из `features` или `support`)
- Предоставляет инфраструктуру, которую используют все вышележащие слои
- UI-компоненты kernel — это дизайн-система, а не бизнес-компоненты

---

### support — вспомогательные бизнес-модули

Переиспользуемые бизнес-модули, которые нужны нескольким фичам. Support знает о домене, но не реализует полноценные пользовательские сценарии.

**Что здесь живёт:**
- Общие доменные хуки (`useMe`, `useMeSuspense`) — запросы к API, которые нужны в разных частях приложения
- Общие доменные типы и утилиты, привязанные к бизнесу

**Правила:**
- Импортирует из `kernel` и `lib`
- Не импортирует из `features`
- Каждый модуль support имеет `index.ts` как публичный API
- Если модуль используется только одной фичей — он должен быть внутри этой фичи, а не в support

---

### features — бизнес-сценарии

Реализация конкретных пользовательских сценариев. Каждая фича — это полноценный use case от данных до UI.

**Что здесь живёт:**
- Полные сценарии: авторизация по OTP, заполнение профиля, и т.д.
- Каждый модуль следует [feature module architecture](feature.md): `domain/`, `model/`, `ui/`, `compose/`, `index.ts`

**Правила:**
- Импортирует из `support`, `kernel` и `lib`
- **Фичи не могут импортировать друг друга напрямую.** Каждая фича изолирована. Если двум фичам нужны общие данные или логика — это выносится в `support`
- Экспортирует compose-компоненты и провайдеры через `index.ts`

---

### app — маршрутизация и точка входа

Expo Router file-based routes. Тонкий слой, соединяющий фичи с навигацией.

**Что здесь живёт:**
- Route-файлы — импортируют compose-компоненты из features и экспортируют как default
- Layout-файлы — настраивают провайдеры, Stack/Tabs навигацию
- Root layout — глобальные провайдеры (API, тема), эффекты (logout listener)

**Правила:**
- Может импортировать из любого слоя
- Файлы максимально тонкие — без бизнес-логики
- Если route нужен роутер-контекст, передаёт колбэки в compose через пропсы

---

## Визуализация

```
┌─────────────────────────────────────────────────┐
│  app                                            │
│  Route-файлы, layouts, глобальные провайдеры     │
├─────────────────────────────────────────────────┤
│  features                                       │
│  Бизнес-сценарии (domain/model/ui/compose)      │
├─────────────────────────────────────────────────┤
│  support                                        │
│  Переиспользуемые бизнес-модули                  │
├─────────────────────────────────────────────────┤
│  kernel                                         │
│  Биндинги lib к проекту, дизайн-система, сессия  │
├─────────────────────────────────────────────────┤
│  lib                                            │
│  Универсальные утилиты без контекста проекта     │
└─────────────────────────────────────────────────┘
```

Каждый слой видит только то, что ниже него.
